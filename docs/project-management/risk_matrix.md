# Matriz de Riesgos Extendida - Proyecto BD2

## 🇪🇸 Matriz de Riesgos del Proyecto (Español)

| ID       | Riesgo Identificado                                                                                                                                                                                                                                               | Categoria         | Probabilidad | Impacto | Estrategia de Mitigación (Plan de Acción)                                                                                                                                             |
| :------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---------------- | :----------- | :------ | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **R-01** | **Inconsistencia de datos entre MySQL y MongoDB (RNF3-02)**<br>Fallas en la sincronización o transacciones huérfanas entre bases de datos.                                                                                                                        | Técnico           | Media        | Alto    | Diseñar un API backend robusto que maneje escrituras controladas y usar UUIDs para cruzar datos.                                                                                      |
| **R-02** | **Cuellos de botella en consultas (RNF2-02)**<br>Filtros combinados de vinos/viñedos superan el límite de 1 segundo.                                                                                                                                              | Técnico           | Alta         | Alto    | Implementar índices en MySQL y evitar agregaciones complejas (`$lookup`) en tiempo real en MongoDB.                                                                                   |
| **R-03** | **Lógica de negocio oculta (RNF3-03)**<br>Abuso de Triggers y Stored Procedures dificulta el mantenimiento.                                                                                                                                                       | Técnico           | Media        | Medio   | Limitar Triggers a auditoría. Centralizar la lógica de negocio compleja en el backend, no en la BD.                                                                                   |
| **R-04** | **Retrasos por interfaces dinámicas (FR2.4, FR3.3)**<br>Crear vistas por rol (entusiasta, experto) consume mucho tiempo.                                                                                                                                          | Técnico           | Alta         | Medio   | Crear un componente de formulario base que renderice campos dinámicamente vía JSON.                                                                                                   |
| **R-05** | **Baja usabilidad en móviles (RNF1-02)**<br>Tablas de datos y métricas se rompen en pantallas pequeñas.                                                                                                                                                           | Técnico           | Baja         | Alto    | Aplicar diseño _Mobile First_ y usar un framework CSS (Tailwind/Bootstrap) con tarjetas apilables.                                                                                    |
| **R-06** | **Incumplimiento o deserción en el equipo**<br>Un miembro no entrega su parte a tiempo o abandona el proyecto, sobrecargando al resto.                                                                                                                            | Equipo            | Media        | Crítico | Hacer reuniones de control cortas (Dailies/Weeklies). Dividir el trabajo en módulos independientes para que el fallo de uno no bloquee todo el sistema.                               |
| **R-07** | **Choque de tiempos con otras entregas**<br>Dejar la integración para la última semana justo cuando hay entregas de otras materias pesadas.                                                                                                                       | Tiempo            | Muy Alta     | Crítico | Establecer fechas límite internas al menos 5 días antes de la entrega real. Integrar el frontend y backend de forma continua, no al final.                                            |
| **R-08** | **Cambios de alcance de última hora (Scope Creep)**<br>El profesor o "cliente" pide nuevos requerimientos que no estaban en el documento original.                                                                                                                | Externo           | Media        | Alto    | Tener los requerimientos base (RNF y FR) firmemente documentados. Negociar que cualquier añadido sea un "plus" o para una versión 2.0.                                                |
| **R-09** | **Conflictos de integración de código (Git Hell)**<br>Al unir las ramas de GitHub el día de la entrega, se sobrescribe código funcional.                                                                                                                          | Equipo / Técnico  | Alta         | Alto    | Prohibir hacer _push_ directo a `main`. Usar _Pull Requests_ obligatorios y que otro miembro revise antes de aprobar (Code Review).                                                   |
| **R-10** | **Curva de aprendizaje técnica**<br>El equipo se estanca aprendiendo a hacer consultas complejas en MongoDB o conectando el ORM.                                                                                                                                  | Tiempo            | Media        | Medio   | Asignar pruebas de concepto tempranas (spikes). Si una tecnología bloquea al equipo más de 2 días, pedir asesoría o buscar una alternativa más simple.                                |
| **R-11** | **Problemas de despliegue (Deployment)**<br>La app funciona en local pero falla al subirse a la nube (servidores, puertos, variables de entorno).                                                                                                                 | Externo / Técnico | Alta         | Alto    | Configurar el entorno de producción (ej. Docker, AWS, Render) en la mitad del proyecto, no la noche anterior a la entrega.                                                            |
| **R-12** | **Pérdida de datos previo a la sustentación**<br>La base de datos de prueba se borra o corrompe accidentalmente horas antes de mostrar el proyecto.                                                                                                               | Técnico           | Baja         | Crítico | Tener un script de _Seed_ (poblado de datos) automatizado que levante la base de datos con información de prueba en segundos.                                                         |
| **R-13** | **Incompatibilidad de entornos de desarrollo y ejecución**<br>Discrepancias en las versiones de dependencias o motores de bases de datos entre las distintas estaciones de trabajo y el equipo de presentación, generando fallos críticos en tiempo de ejecución. | Técnico           | Alta         | Alto    | Estandarizar el entorno mediante la contenerización de servicios con Docker (Docker-compose) y el uso de gestores de versiones de dependencias (ej. `.nvmrc`).                        |
| **R-14** | **Sobrecarga por cruce de parciales/proyectos**<br>Las fechas de entrega coinciden exactamente con proyectos pesados de Sistemas Operativos, Dinámica o Complejidad, dejando al equipo sin tiempo físico.                                                         | Tiempo            | Muy Alta     | Crítico | Adelantar el esqueleto del backend y las conexiones a BD en las semanas de menor carga. Dejar estrictamente solo retoques visuales del frontend para la "semana del terror".          |
| **R-15** | **Saturación de conexiones a la Base de Datos (Connection Pool)**<br>Como los dashboards consultan muchas métricas (FR1.1), recargar la página varias veces agota el límite de conexiones gratuitas en la nube y tumba la app.                                    | Técnico           | Media        | Alto    | Implementar un patrón _Singleton_ para reutilizar la misma conexión a la base de datos en el backend, e implementar un sistema básico de caché en memoria para las métricas globales. |

## 🇬🇧 Project Risk Matrix (English)

| ID       | Identified Risk                                                                                                                                                                                                            | Category    | Probability | Impact   | Mitigation Strategy (Action Plan)                                                                                                                   |
| :------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---------- | :---------- | :------- | :-------------------------------------------------------------------------------------------------------------------------------------------------- |
| **R-01** | **Data inconsistency (MySQL & MongoDB) (RNF3-02)**<br>Sync failures or orphaned transactions between databases.                                                                                                            | Technical   | Medium      | High     | Design a robust backend API for controlled writes and use UUIDs for data referencing.                                                               |
| **R-02** | **Query performance bottlenecks (RNF2-02)**<br>Combined filtering operations exceed the 1-second threshold.                                                                                                                | Technical   | High        | High     | Implement database indexing in MySQL and avoid real-time heavy aggregations in MongoDB.                                                             |
| **R-03** | **Hidden business logic (RNF3-03)**<br>Overuse of Triggers and Stored Procedures hinders maintenance.                                                                                                                      | Technical   | Medium      | Medium   | Restrict Triggers to auditing. Centralize complex business logic in the backend layer.                                                              |
| **R-04** | **Dynamic forms complexity delays (FR2.4, FR3.3)**<br>Building role-based views consumes excessive development time.                                                                                                       | Technical   | High        | Medium   | Build a reusable base form component that dynamically renders fields via JSON config.                                                               |
| **R-05** | **Mobile usability degradation (RNF1-02)**<br>Data tables and metrics break on small screens.                                                                                                                              | Technical   | Low         | High     | Apply a Mobile First approach using a CSS framework with stackable cards instead of wide tables.                                                    |
| **R-06** | **Team member underperformance or dropout**<br>A team member fails to deliver or leaves, overloading the rest.                                                                                                             | Team        | Medium      | Critical | Hold short sync meetings. Modularize tasks so one person's delay doesn't block the entire core system.                                              |
| **R-07** | **Time crunch & schedule clashes**<br>Leaving integration for the final week, colliding with exams or other heavy subjects.                                                                                                | Time        | Very High   | Critical | Set internal deadlines at least 5 days before the actual due date. Practice continuous integration instead of Big Bang integration.                 |
| **R-08** | **Last-minute scope creep**<br>The professor/client requests new features not included in the original specs.                                                                                                              | External    | Medium      | High     | Strictly document base requirements (FRs/NFRs). Negotiate additions as "nice-to-have" features for future iterations.                               |
| **R-09** | **Code integration conflicts (Git Hell)**<br>Merging branches on delivery day overwrites functional code.                                                                                                                  | Team / Tech | High        | High     | Forbid direct commits to `main`. Enforce mandatory Pull Requests and peer Code Reviews before merging.                                              |
| **R-10** | **Steep technical learning curve**<br>The team gets stuck learning MongoDB aggregations or ORM setups.                                                                                                                     | Time        | Medium      | Medium   | Assign early proofs of concept (spikes). If stuck for over 48 hours, seek mentorship or pivot to a simpler approach.                                |
| **R-11** | **Deployment and cloud hosting failures**<br>App works locally but crashes on the cloud (env variables, ports).                                                                                                            | Ext / Tech  | High        | High     | Set up the production environment (e.g., Docker, AWS, Render) mid-project, not the night before the deadline.                                       |
| **R-12** | **Pre-presentation data loss**<br>The test database is accidentally wiped or corrupted hours before the demo.                                                                                                              | Technical   | Low         | Critical | Create an automated Database Seed script to instantly repopulate the database with realistic mock data.                                             |
| **R-13** | **Development and Execution Environment Incompatibility**<br>Discrepancies in dependency versions or database engines between development workstations and the presentation machine, leading to critical runtime failures. | Technical   | High        | High     | Standardize the environment by containerizing services using Docker (Docker-compose) and implementing dependency version managers (e.g., `.nvmrc`). |
| **R-14** | **Overload from overlapping projects/midterms**<br>Delivery dates coincide exactly with heavy Operating Systems, Dynamics, or Complexity projects, leaving no physical time.                                               | Time        | Very High   | Critical | Push backend scaffolding and DB connections during low-load weeks. Strictly reserve the "hell week" only for minor frontend visual tweaks.          |
| **R-15** | **Database Connection Pool Exhaustion**<br>Due to multiple metric queries (FR1.1), repeated page reloads hit the cloud free-tier connection limits, crashing the app.                                                      | Technical   | Medium      | High     | Implement a _Singleton_ pattern in the backend to reuse the database connection, and add basic in-memory caching for global aggregate metrics.      |
