## Description

**Statement:**
> The vineyard page must allow the user to create a review of the vineyard through a form that varies according to the selected reviewer type: amateur, expert or organization.

**Involved Actor:**
Any visitor to the vineyard page. No authentication is required. The reviewer type is selected by the user from the form, without formal identity verification.

---

## Conditions

**Precondition:**
- The user is on a vineyard's page.
- The review creation form is visible or accessible from the vineyard's reviews section.

**Postcondition:**
The review is stored in MongoDB as a document with `vineyard_id` corresponding to the current vineyard and `reviewer_type` according to the user's selection. The vineyard's metrics in `vineyard_metrics` are updated by the backend upon receiving the new review.

---

## Acceptance Criteria

### Reviewer type selection
- [ ] **AC1:** The form presents the user with three reviewer type options: **Amateur**, **Expert** or **Organization** before showing the specific fields.
- [ ] **AC2:** When selecting the reviewer type, the form shows only the fields corresponding to that profile.
- [ ] **AC3:** The reviewer type cannot be modified once the form filling has started without restarting it.

### Form — Amateur
- [ ] **AC4:** The amateur form includes the following mandatory fields: reviewer name (`reviewer_name`), visit date (`visit_date`), overall rating (`score_overall`, scale 0–100), experience description (`experience_description`) and recommendation indicator (`would_recommend`, yes/no).

### Form — Expert and Organization
- [ ] **AC5:** The expert or organization form includes the following mandatory fields: reviewer name (`reviewer_name`), occupation (`occupation`), organization they belong to (`organization`), years of experience (`years_experience`), visit date (`visit_date`), overall rating (`score_overall`, scale 0–100), tasting notes (`tasting_notes`) and pairing suggestions (`pairing_suggestions`).

### Validations
- [ ] **AC6:** All form fields are mandatory — the system prevents submission if any is empty, showing an error message per missing field.
- [ ] **AC7:** The `score_overall` field accepts only numeric values between 0 and 100. Values outside the range are rejected with an error message.
- [ ] **AC8:** The `years_experience` field accepts only positive integer values greater than 0.
- [ ] **AC9:** The `visit_date` field cannot be a future date.

### Submission and confirmation
- [ ] **AC10:** When the form is submitted correctly, the system shows a confirmation message and the review appears in the vineyard's listing without needing to reload the page.
- [ ] **AC11:** If an error occurs when saving the review, the system shows a descriptive error message and preserves the entered data so the user can retry.
- [ ] **AC12:** The document stored in MongoDB includes the `submitted_at` field automatically generated by the backend with the date and time of submission — this field is not visible or editable by the user.

## Notes and Change History

- Vineyard reviews never include `wine_id` — they are exclusive to the vineyard as an entity.
- `submitted_at` is generated by the backend at the time of saving, not by the frontend.
- The backend is responsible for updating `vineyard_metrics` in MySQL after receiving and storing a new review in MongoDB.
- There is no reviewer identity verification — the system trusts the honest selection of reviewer type by the user.
